# Onboarding Tutorial Findings

- Game feel over form: the pre-TUI wizard frames auth as stocking a station, leads with the mascot, highlights `Demo Kitchen` as the no-account tasting-menu path, and ends with the exact first move: `What's my quest?`. The TUI then keeps the same teaching voice with a mascot header, active placeholder, and `NEXT UP` line instead of making the player infer commands from check text.
- Wizard/TUI seam: onboarding stays text-mode ANSI on purpose. It runs before OpenTUI so OAuth/browser/device-code prompts cannot fight the TUI renderer or approval modal input focus. It hands one small `{ provider, method, account }` auth result to `wireHarness`; the TUI only consumes the resulting quest state and hints.
- Front-matter hint limitation for LOO-174: the v1 quest loader validates markdown front matter with `QuestSchema` as a strict object, so adding an unknown `hint` key currently fails pack loading rather than flowing through `QuestView`. The prototype therefore keeps hints in `proto/tui/questlog.tsx` as `HINTS: Record<questId, string>` and leaves the desired front-matter field as a loader/schema follow-up.
- LOO-169/174/179 re-scope input: the init wizard is LOO-169 territory now because founder feedback made subscription OAuth a tutorial beat, not just provider setup. LOO-174 should own durable quest hint schema/threading. LOO-179 should treat guided prompt copy as part of game feel alongside retro presentation, not as generic form help.
